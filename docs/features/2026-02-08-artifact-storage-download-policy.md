# Artifact Storage and Download Policy

## Status

`ready`

## Problem

After builds complete, operators need to download produced artifacts (APKs, IPAs, macOS apps) for testing and distribution. Without a structured artifact storage and download mechanism, build outputs are ephemeral — lost when the runner cleans up its workspace. Artifacts must be stored durably, associated with their build, and accessible via time-limited links that prevent unauthorized long-term access to binaries.

## User Impact

- **Developers** can download build artifacts directly from the build detail page without SSH-ing into runners or navigating file systems.
- **QA engineers** can retrieve the exact artifact for a specific build to test, with confidence that the checksum matches what the runner produced.
- **Admins and operators** can audit who generated download links and when, providing traceability for artifact access.
- Download links expire after 15 minutes for security. Authorized users can re-generate links as needed — there is no limit on how many times a link can be re-issued, only on how long each link remains valid.

## UI Changes

The build detail page now includes an **Artifacts** section visible when a build has reached a terminal state (succeeded or failed) and has registered artifacts:

- **Artifact list** displays each artifact's name, type (e.g., APK, IPA, macOS App), file size (human-readable), and SHA-256 checksum (truncated with copy-to-clipboard).
- **Download button** on each artifact row generates a time-limited download link via the API and opens it in a new tab. If storage is disabled, the button shows an error state with a descriptive message.
- **Empty state** when a build has no artifacts displays a message indicating no artifacts were produced.
- For builds still in progress, the artifacts section shows a placeholder indicating artifacts will appear after the build completes.

## API Changes

New and existing endpoints:

- `POST /v1/runners/{runner_id}/jobs/{job_id}/artifacts` — Runner registers an artifact manifest after producing a build output. Payload includes `name` (string), `artifact_type` (string, one of: "apk", "ipa", "app", "generic"), `checksum` (optional SHA-256 hex string), `file_size` (optional integer, bytes), and `metadata` (optional JSON object). Authenticated with runner token (RunnerAuth). Returns 200 with `{ "artifact": {...}, "upload_url": "..." }`. The `upload_url` is generated by the active storage backend: pre-signed S3-compatible PUT URL, or local signed upload URL. If storage is disabled, `upload_url` is empty.
- `GET /v1/builds/{build_id}/artifacts` — List all artifacts for a build. Returns `{ "artifacts": [...] }` where each artifact includes id, build_id, name, artifact_type, file_path, file_size, checksum, metadata, and created_at. Requires AuthUser with RBAC permission `builds:read`.
- `POST /v1/artifacts/{artifact_id}/download-link` — Generate a pre-signed download URL for the artifact. The URL is valid for 15 minutes. Returns `{ "download_url": "...", "expires_at": <unix_timestamp> }`. Requires AuthUser with RBAC permission `builds:read`. Each invocation generates a new audit log entry. Returns 503 if artifact storage is disabled.

Existing database schema:

- The `artifacts` table was created in migration 005 with columns for `id`, `build_id`, `name`, `artifact_type`, `file_path` (storage key), `file_size`, `checksum`, `metadata` (JSON), `created_at`. Storage settings persistence is added by migration 011.

## Security Considerations

- **Signed URLs with 15-minute TTL** — Download URLs are generated server-side using the active backend (S3-compatible presigned URLs or local signed URLs). Expired links fail safely and are re-issuable via API.
- **RBAC enforcement** — The `builds:read` permission is required before any download link is generated. Users without this permission cannot access artifact metadata or obtain download URLs.
- **Audit logging** — Every call to `POST /v1/artifacts/{artifact_id}/download-link` creates an `audit_logs` entry recording the user, artifact, and timestamp. This provides a complete trail of who accessed which artifacts.
- **Runner authentication** — Artifact registration is authenticated via the runner token with the same cross-runner ownership checks used by other runner endpoints (runner can only register artifacts for builds it claimed).
- **Storage isolation** — For S3/R2, buckets should be private-only and all access should go through signed URLs generated by the daemon. For local backend, files are stored under the configured base directory and accessed via short-lived signed daemon URLs.
- **Integrity verification** — The SHA-256 checksum recorded at upload time allows consumers to verify artifact integrity after download. The checksum is displayed in the UI and returned in the API response.

## Migration and Rollout

Artifact storage can be configured through Settings UI/API (`/v1/settings/artifact-storage`) with providers `disabled`, `local`, `s3`, `r2`. Environment variables remain supported as fallback when DB settings are absent:

- `OORE_S3_BUCKET` (required) — bucket name for artifact storage
- `AWS_ACCESS_KEY_ID` (required) — S3 access key (standard AWS credential env var)
- `AWS_SECRET_ACCESS_KEY` (required) — S3 secret key (standard AWS credential env var)
- `OORE_S3_ENDPOINT` (optional) — S3-compatible endpoint URL (supports MinIO, Cloudflare R2, AWS S3, etc.)
- `OORE_S3_REGION` (optional, default: `us-east-1`) — storage region

When storage is disabled, artifact metadata is still persisted in the database but `upload_url` is empty and download link generation returns 503 Service Unavailable. Builds still execute normally — only binary artifact transfer is unavailable.
