# Setup Wizard UI

## Status

`ready`

## Problem

Operators need a guided, web-based interface to configure a fresh oore.build instance. Without a UI, setup is limited to the CLI or raw API calls, which increases the barrier to onboarding. The wizard must enforce the same state-machine ordering as the backend, handle session expiry gracefully, and support resuming from any intermediate state.

## User Impact

Operators can now navigate to `/setup` in their browser to complete instance configuration through a 4-step wizard. The wizard mirrors the CLI interactive flow but provides a visual interface with inline validation, real-time session countdown, and structured error messages. After completion, the operator is directed to the main dashboard.

## UI Changes

### Setup Layout (`/setup`)

The `/setup` route renders a centered card layout with:

- **Title and subtitle**: "oore.build Setup" / "Configure your self-hosted CI instance".
- **Step indicator**: Horizontal badge bar showing Token, OIDC, Owner, Complete. Active step uses `default` variant, completed steps show a checkmark with `secondary` variant, future steps use `outline`.
- **Session countdown**: Displays remaining time as `MM:SS` in monospace font. Turns red (`text-destructive`) when under 5 minutes. On expiry, resets the Zustand store and redirects to `/setup`.
- **Route guard**: The `beforeLoad` hook fetches `/v1/public/setup-status`. If `is_configured` is true, redirects to `/` (dashboard).

### Step 1: Bootstrap Token (`/setup/`)

- Password input for the bootstrap token (generated by `oore setup open`).
- Submits to `POST /v1/setup/bootstrap-token/verify`.
- On success, stores `session_token` and `expires_at` in Zustand (backed by `sessionStorage`), advances to step 2.
- Displays specific error messages for expired, consumed, and invalid tokens.
- Auto-advances to the correct step if the backend is already past bootstrap and a valid session exists.

### Step 2: OIDC Configuration (`/setup/oidc`)

- Three input fields: Issuer URL (required, type=url), Client ID (required), Client Secret (optional, type=password).
- Route guard redirects to `/setup` if no session token.
- Submits to `POST /v1/setup/oidc/configure` with Bearer auth.
- On success, shows a discovery confirmation alert with the verified issuer URL, then auto-advances after 1.2 seconds.
- Warning text: "Double-check your values before submitting -- this step cannot be undone."
- Inputs are disabled after successful submission to prevent double-submit.

### Step 3: Owner Account (`/setup/owner`)

- Email input with client-side validation (`/^[^\s@]+@[^\s@]+\.[^\s@]+$/`).
- Route guard redirects to `/setup` if no session token.
- Submits to `POST /v1/setup/owner/finalize` with Bearer auth.
- Warning text: "Make sure this matches your OIDC provider email -- this cannot be changed after submission."
- Inline validation error shown below the input when the email format is invalid.

### Step 4: Complete Setup (`/setup/complete`)

- Configuration summary showing current state and truncated instance ID.
- Destructive alert warning that completing setup permanently locks all setup endpoints.
- Route guard redirects to `/setup` if no session token.
- Submits to `POST /v1/setup/complete` with Bearer auth.
- On success, clears session token from store, advances step to 4 (all steps show completed), displays instance ID, and shows a "Go to Dashboard" link.

### State Management

- **Zustand store** (`useSetupStore`): Tracks `currentStep`, `sessionToken`, and `sessionExpiresAt`. Session data is persisted to `sessionStorage` under `oore_setup_session` and `oore_setup_session_expires` keys. The `reset()` method clears all state.
- **TanStack Query hooks** (`use-setup.ts`): `useSetupStatus` polls `/v1/public/setup-status` every 3 seconds. Mutation hooks (`useVerifyBootstrapToken`, `useConfigureOidc`, `useFinalizeOwner`, `useCompleteSetup`) invalidate the status query on success.
- **Session countdown hook** (`useSessionCountdown`): 1-second interval timer that computes remaining seconds from `sessionExpiresAt`. Returns `formatted` (MM:SS), `isWarning` (< 5 min), and `isExpired` (0 seconds remaining).

### API Client

The `api.ts` module provides a typed fetch wrapper with structured error handling. Non-2xx responses are parsed as `ApiError` objects and thrown as `ApiClientError` instances carrying `status`, `code`, and `details` fields for granular error display in the UI.

## API Changes

No new API endpoints. The wizard consumes the existing setup endpoints documented in the setup bootstrap foundation feature doc:

- `GET /v1/public/setup-status`
- `POST /v1/setup/bootstrap-token/verify`
- `POST /v1/setup/oidc/configure`
- `POST /v1/setup/owner/finalize`
- `POST /v1/setup/complete`

## Security Considerations

- **Session token storage**: The session token is stored in `sessionStorage` (not `localStorage`), scoping it to the browser tab and clearing it on tab close.
- **Route guards**: TanStack Router `beforeLoad` guards prevent accessing setup steps without a valid session token in the Zustand store.
- **Setup-complete redirect**: The parent `/setup` route checks `is_configured` on every load and redirects to `/` if setup is already complete, preventing re-entry.
- **Session expiry auto-reset**: The countdown hook triggers a full store reset and redirect on expiry, preventing stale session tokens from being sent to the API.
- **No secrets in client state**: The bootstrap token is not stored after verification. Only the session token (which is opaque and hashed server-side) is kept in the store.
- **Bearer auth**: All mutating setup requests include the session token as a Bearer token in the Authorization header via the `authHeaders` utility.

## Migration and Rollout

First implementation. The `/setup` route is only accessible when the instance is not yet configured. Once setup completes, all visits to `/setup` redirect to the dashboard.
