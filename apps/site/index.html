Installation details

From revision to verified artifact

Oore keeps the workflow visible while the build stays on macOS hardware you operate.

  1. Approve the source

    Connect GitHub or GitLab, then approve each repository this runner may execute.

  2. Trigger a revision

    Use verified pushes, same-repository pull requests, manual builds, reruns, or the API.

  3. Build in a private workspace

    The Direct runner checks out the exact revision and runs your Flutter toolchain on macOS.

  4. Sign, verify, and distribute

    Runner-owned signing logic verifies output before artifacts are stored and shared.

Operate without living in SSH

Projects, queues, logs, artifacts, users, sources, and runner policy stay in one web interface.

Direct execution, clearly bounded

Direct mode prioritizes macOS toolchain compatibility. It is not a hostile-code sandbox.

Repository authority

Build commands run with the permissions of the runner's macOS account.

  • Owners and admins enable the instance and approve each repository.
  • External-fork pull and merge requests are ignored.
  • A dedicated non-admin runner account is recommended.

Signing hygiene

Repository commands do not receive signing files, passwords, grant tokens, or an unlocked keychain.

  • Credentials are encrypted at rest.
  • Signing material is retrieved late with a job-scoped grant.
  • Runner-owned logic signs, verifies, and cleans temporary state.

These controls reduce accidental exposure. They do not contain malicious code running as the same macOS account.

Install once. Update from the UI.

The managed updater verifies the release, installs it, and restarts the daemon, local web service, and Direct runner after the backend toolchain is ready.

A normal managed installation does not require SSH or a separate launchctl kickstart after an update. Foreground and otherwise unmanaged processes remain operator-managed.
