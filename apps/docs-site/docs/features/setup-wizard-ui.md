# Setup Wizard UI

The web-based setup wizard provides a guided 4-step interface for configuring a fresh oore.build instance. It is available at `/setup` and mirrors the [CLI interactive flow](/cli/setup).

## Overview

The wizard walks the operator through the full [setup state machine](/features/setup-wizard#state-machine):

| Step | Route | Action |
|---|---|---|
| 1. Token | `/setup/` | Verify bootstrap token, acquire session |
| 2. OIDC | `/setup/oidc` | Configure identity provider |
| 3. Owner | `/setup/owner` | Verify owner identity via OIDC |
| 4. Complete | `/setup/complete` | Finalize and lock setup endpoints |

## Step Indicator

A horizontal badge bar at the top of the wizard shows progress:

- **Active step**: `default` badge variant
- **Completed steps**: `secondary` badge with checkmark
- **Future steps**: `outline` badge

## Session Countdown

A real-time countdown timer displays the remaining setup session time in `MM:SS` format:

- Updates every second
- Turns red (`text-destructive`) when under 5 minutes remaining
- On expiry: resets the store and redirects to `/setup`

## Route Guards

Each step beyond Token uses a TanStack Router `beforeLoad` guard:

- If no session token is present in the Zustand store, the guard redirects to `/setup`
- The parent `/setup` route checks `is_configured` from the setup status API -- if `true`, it redirects to `/` (dashboard)

## Step Details

### Step 1: Bootstrap Token

- Password input for the token generated by [`oore setup open`](/cli/setup#setup-open)
- Submits to [`POST /v1/setup/bootstrap-token/verify`](/api/setup#verify-bootstrap-token)
- On success: stores `session_token` and `expires_at` in Zustand, advances to step 2
- Displays specific error messages for:
  - Expired tokens (`token_expired`)
  - Already-consumed tokens (`token_consumed`)
  - Invalid tokens (`invalid_token`)
- **State resumption**: If the backend is already past bootstrap, the token step detects this and auto-advances

### Step 2: OIDC Configuration

- Three input fields: Issuer URL (required), Client ID (required), Client Secret (optional)
- Submits to [`POST /v1/setup/oidc/configure`](/api/setup#configure-oidc)
- On success: shows a discovery confirmation with the verified issuer URL, then auto-advances after 1.2 seconds
- Inputs are disabled after successful submission to prevent double-submit

::: warning
A warning is shown: "Double-check your values before submitting -- this step cannot be undone."
:::

### Step 3: Owner Account

- Email input with client-side validation (`/^[^\s@]+@[^\s@]+\.[^\s@]+$/`)
- Initiates OIDC flow via [`POST /v1/setup/owner/start-oidc`](/api/setup#start-owner-oidc), then verifies via [`POST /v1/setup/owner/verify-oidc`](/api/setup#verify-owner-oidc)
- Inline validation error shown when format is invalid

::: warning
Warning text is displayed: "Make sure this matches your OIDC provider email -- this cannot be changed after submission."
:::

### Step 4: Complete

- Configuration summary showing current state and truncated instance ID
- Destructive alert warning that completing setup permanently locks all setup endpoints
- Submits to [`POST /v1/setup/complete`](/api/setup#complete-setup)
- On success: clears session token, marks all steps complete, displays instance ID and "Go to Dashboard" link

## State Management

### Zustand Store (`useSetupStore`)

| Field | Type | Description |
|---|---|---|
| `currentStep` | `number` | Active wizard step (0-3) |
| `sessionToken` | `string \| null` | Setup session token |
| `sessionExpiresAt` | `number \| null` | Session expiry (Unix epoch) |

Session data is persisted to `sessionStorage` under:
- `oore_setup_session` -- session token
- `oore_setup_session_expires` -- expiry timestamp

The `reset()` method clears all state and removes `sessionStorage` entries.

### TanStack Query Hooks

| Hook | Endpoint | Behavior |
|---|---|---|
| `useSetupStatus` | `GET /v1/public/setup-status` | Polls every 3 seconds |
| `useVerifyBootstrapToken` | `POST /v1/setup/bootstrap-token/verify` | Invalidates status on success |
| `useConfigureOidc` | `POST /v1/setup/oidc/configure` | Invalidates status on success |
| `useSetupOidcStart` | `POST /v1/setup/owner/start-oidc` | Starts OIDC flow for owner verification |
| `useSetupOidcVerify` | `POST /v1/setup/owner/verify-oidc` | Invalidates status on success |
| `useCompleteSetup` | `POST /v1/setup/complete` | Invalidates status on success |

### Session Countdown Hook (`useSessionCountdown`)

Returns:
- `formatted` -- remaining time as `MM:SS`
- `isWarning` -- `true` when under 5 minutes
- `isExpired` -- `true` when time has reached zero

## Security Considerations

- **Session token in `sessionStorage`**: Scoped to the browser tab, cleared on tab close (not `localStorage`)
- **Route guards**: Prevent accessing later steps without authentication
- **Setup-complete redirect**: Prevents re-entry after setup is finished
- **Session expiry auto-reset**: Prevents stale tokens from being used
- **No secrets in client state**: Bootstrap token is not stored after verification; only the opaque session token is kept
- **Bearer auth**: All mutating requests use the session token as a Bearer token in the Authorization header
